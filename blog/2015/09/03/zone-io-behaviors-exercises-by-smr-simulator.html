Notes on Zone I/O Behaviors and Exercises by SMR Simulator

Overview

SMR Simulator (SMRSim) is a software tool and simulation on host managed SMR storage drive which enforces the core ZAC/ZBC rules defined by the INCITS technical T10/T13 committees.

By releasing the SMR simulator, we hope to enable open source developers experiment and become familiar with SMR functionality and behaviors without the need to access a real SMR (ZBC/ZAC) hardware. SMR simulator keeps statistics on number of rule violations in each zone. When the SMR storage is mounted to a standard file system (not SMR aware) to obtain write through rule violation statistics over the whole capacity are valuable in estimating the overall conditions. Usage status over the SMR zone are observable to both user applications and file systems. Getting violation counts will help file system and application developers identify what needs to be done to make their SW SMR-friendly.

Default Behavior Expectations

Rules for out of policy read: (1) Starts or goes beyond WP (Write Pointer); (2) Spans zones. Rules for out of policy write: (1) Does not start on WP; (2) Spans zones; (3) Is not 4k aligned in SMR zones.

By default, if a host sends an out of policy read/write then SMRSim will record each rule violation counter, reject the operation, and return an error.

Behavior Benchmark and Extension

The default expectations on SMR zone access behaviors are soon challenged when mounting the simulator volume to a standard file system and passing through read/write operations. How far these operations can go and how many violations happens in each zone can not be obtained as the earlier rule violation will discontinue any further operation. As a software simulation, new options can always be added to measure the relevant data in order to progressively produce better solutions.

With behavior measuring extensions, SMRSim is capable of being configured to do full pass-through read/write over the entire storage volume without failure but recording all rule violations and zone status. With such option, the error statistics and usage status in each zone can be reviewed and assessed. The view across all zones give a clear picture of where to focus on to solve issues. For example, if the most of the "write not at write-pointer" errors are happening near the beginning of the zones then we can configure these zones as the CMR zones. This will resolve the write errors without breaking the SMR standard.

With the “SMR Zone Across” option, if the incoming data size is bigger than a zone size when the contiguous zones are empty then it can be split up to fit into two zones without violating the rules.

Simplified Error Feedback Path

Error handling has always been a complex issue. According to the standard, error codes will need go through sense code mechanisms like what SCSI error propagation does. The distribution of the error retrieval mechanism to the user application or file systems is a non-trivial task.

SMRSim is built on top of the Linux Kernel Device Mapper framework. The potential extension on the framework will allow the plug-in module to propagate the error code back to the upper layer. Therefore, there are opportunities for the plug-in module to be a translation point and "return" the appropriate error to the user.

Host Managed SMR Device Mapper

SMRSim provides the basic functionality and configuration options to support an I/O indirection layer built on top of the device mapper framework to handle the host managed tasks. As a supporting tool, it enables Host Managed SMR Device Mapper (SMR-DM) to address more area yet need to be discovered from the real HW limitations. SMR-DM is designed to wrap up the bare metal drive which has simplified SMR firmware and leave the flexibility and sensibility out of the box. To simplify the error feedback path, the device mapper plugin layer can translate the "sense codes" into functional errors and return this to the user. For more introduction details, refer to article [1].

Conclusion

SMRSim is a simple tool to simulate SMR drive behaviors with flexible extensions. We hope it is useful to the community to further develop applications or system software and adapt to upcoming high density SMR drives. The incoming SMR Device Mapper module will be the next power component on the host managed SMR solutions.
